# SelfSync — optional client-side E2E encryption (design, for review)

**Date:** 2026-07-05 · **Status:** DESIGN — not implemented; needs review + a decision
(E2EE was deferred in D0007). **Why design-first:** shipping a cryptographic protocol blind
is a security liability that can't be caught in casual testing — it warrants your review.

## Goal

An **optional** mode where note content is encrypted on the client so a compromised or
malicious server (or anyone with disk access) cannot read it — closing the one gap in the
current trusted-server model. Off by default; opt-in per vault.

## What gets encrypted

- **Chunk content — yes.** Each content chunk is encrypted client-side before upload; the
  server only ever sees ciphertext.
- **Paths / filenames — NO (recommended).** LiveSync obfuscates paths; we deliberately would
  not, to preserve SelfSync's "real browsable files in a bind mount" property (a core value).
  Consequence: the server learns file *names/sizes/structure*, not *contents*. This is an
  explicit, documented trade-off — flag if you want path obfuscation instead (it breaks the
  readable bind mount).

## Key management

- A separate **encryption password** (distinct from the login password), à la Obsidian Sync.
  The setup wizard already leaves room for this step (D0007).
- Derive a symmetric key: **Argon2id(encryption password, per-vault salt)**. The salt is
  non-secret, stored on the server per vault. The encryption password is **never** sent to the
  server and is not recoverable — lose it, lose the data (documented, like Obsidian Sync).
- Cipher: **XChaCha20-Poly1305** (AEAD; misuse-resistant nonce size). Client crypto via
  WebCrypto where possible; XChaCha may need a small audited WASM/JS lib (CSP + bundle impact
  to weigh — a sub-decision).

## The dedup ⇄ security trade-off (the key decision)

Chunks are content-addressed by hash. Two options:

1. **Convergent encryption** (nonce derived from the plaintext hash): identical plaintext →
   identical ciphertext → **dedup preserved**, but it *reveals which chunks are equal* (a known
   convergent-encryption leak). 
2. **Random nonce per chunk**: no equality leak, but **identical plaintext no longer dedups**
   (storage/transfer grows).

Recommendation: **convergent within a single (vault, key) scope** — the equality leak is to a
server that already sees your file structure, and dedup matters for the product's efficiency
goal. Decide explicitly.

## Protocol touch-points

- Client `chunk()` → encrypt each chunk → address by ciphertext hash; `fetchFileBytes` →
  decrypt after fetch. The server is unchanged (it stores opaque bytes).
- A per-vault `encrypted: true` marker + salt (new small server metadata, or a reserved file).
- `commit` file hash: keep the plaintext SHA for the client's own reconcile/base, but the
  server stores the ciphertext chunk list.

## Migration

E2EE can't retroactively encrypt an existing plaintext vault without re-upload. Options:
opt-in **at vault creation**, or an explicit **"encrypt this vault"** operation that re-uploads
all content under the new key. A vault is all-or-nothing encrypted.

## Open decisions for you

1. Content-only encryption (paths visible, bind mount stays readable) — or path obfuscation too?
2. Convergent (dedup, equality leak) vs random-nonce (no dedup) chunk encryption?
3. Cipher/lib: accept a WASM/JS AEAD lib (CSP + bundle cost) for XChaCha, or use WebCrypto
   AES-256-GCM (native, but 96-bit nonce → convergent nonce needs care)?
4. Ship E2EE as its own release, gated behind an explicit opt-in, with a bold "lose the
   password = lose the data" warning?

Nothing here is built yet — this is the artifact to review before implementation.
